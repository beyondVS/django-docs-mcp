# 기능 명세서: 고성능 하이브리드 검색 및 Late Interaction 리랭커 도입

**기능 브랜치**: `005-optimize-search-rerank`
**생성일**: 2026-03-20
**상태**: 초안(Draft)
**입력**: 사용자 설명: "현재 1.5초가 소요되는 느린 문서 검색 속도를 1초 이내로 단축하면서도, 검색 결과의 정밀도를 최고 수준(Cross-Encoder급)으로 유지하는 고성능 하이브리드 검색 엔진을 구축하고자 한다. 이를 위해 다음과 같은 핵심 전략을 명세에 포함한다: 1. 단일 모델 기반 통합 추론: - 'gpahal/bge-m3-onnx-int8' 모델 하나를 사용하여 1차 검색용 임베딩과 2차 리랭킹용 멀티 벡터 생성을 모두 처리한다. - 검색 시 단 한 번의 모델 호출로 두 가지 데이터를 동시에 확보하여 연산 효율을 극대화한다. 2. 하이브리드 검색 고도화: - 기존 RAW SQL로 구현되었던 하이브리드 검색을 'django-paradedb'를 활용하여 django orm 스타일의 BM25와 1,024차원 Dense 벡터 검색을 결합한 RRF 기반 하이브리드 검색으로 구현한다. - 1차 검색 후보군을 50개로 확대하여 리랭킹의 정밀도를 높인다. 3. Late Interaction 리랭커 도입: - 문서 본문의 실시간 재임베딩 오버헤드를 제거하기 위해 Late Interaction(ColBERT-style) 아키텍처를 도입한다. - gpahal/bge-m3-onnx-int8 모델로 멀티벡터를 만들때 1,024차원을 128차원으로 축소(Matryoshka), 토큰 길이는 512로 하여 저장한다. - 압축된 멀티 벡터와 질문 멀티 벡터 간의 MaxSim(최대 유사도 합) 연산을 통해 최종 순위를 산출한다. 4. 시스템 무결성 및 성능: - 모든 연산은 CPU 환경의 ONNX Runtime에서 최적화하며, 저장 공간(DB 용량)과 응답 속도(Latency) 사이의 최적의 균형점을 찾는다. - 사용자가 질문을 입력하자마자 '즉각적인' 검색 결과를 얻을 수 있는 쾌적한 검색 경험을 제공한다."

## Clarifications

### Session 2026-03-20
- Q: 기존 문서 및 신규 문서에 대한 멀티 벡터 생성 전략은? → A: 문서 수집(Ingestion) 명령을 통해 임베딩을 수행할 때 멀티 벡터를 함께 생성 및 구축함.
- Q: 리랭킹용 멀티 벡터의 물리적 저장 방식은? → A: Postgres의 bytea 컬럼에 바이너리 형태로 직접 저장하여 관리 포인트를 통합함.
- Q: 리랭킹 시 실시간 임베딩을 수행하는가? → A: 아니오. 1초 이내 응답 속도 확보를 위해 인제션 시점에 미리 계산된 멀티 벡터를 DB에서 로드하여 사용함.
- Q: 문서 수정 시 데이터 동기화는 어떻게 처리하는가? → A: 한 번 인제션된 문서는 수정되지 않는다고 가정(Immutable)하여 복잡한 동기화 로직을 배제함.
- Q: 멀티 벡터 저장 시 패딩 토큰을 포함할 것인가? → A: 아니오. 패딩(Padding) 토큰을 제외한 실제 유효 토큰 벡터만 가변 길이로 저장하여 저장 용량 및 I/O 효율을 최적화함.
- Q: 멀티 벡터의 세부 바이너리 저장 포맷은? → A: `[Token Count (2B)][Vector Data (N * 128B)]` 구조로 저장하여, 로드 시 즉각적인 벡터 복원과 가변 길이 대응이 가능하도록 함.
- Q: 리랭킹 점수(MaxSim) 정규화 및 최종 점수(relevance_score) 산출 방식은? → A: 쿼리 길이 정규화(Mean MaxSim, `Total_MaxSim / Query_Token_Count`) 방식을 사용하여 최종 relevance_score를 0.0~1.0 범위로 산출함.
- Q: 리랭킹 단계에서 기술적 오류가 발생할 경우 폴백 전략은? → A: 1차 하이브리드 검색(RRF) 결과를 그대로 반환하고 메타데이터에 관련 사실을 기록함.
- Q: 리랭킹 결과 모든 후보의 점수가 임계값(0.3) 미만인 경우의 처리는? → A: 관련성 있는 결과가 없는 것으로 판단하여 빈 결과를 반환하고 사용자에게 안내함.
- Q: 멀티 벡터 저장 시 특수 토큰([CLS], [SEP] 등)을 포함할 것인가? → A: 예. 문맥 정보 유지를 위해 패딩을 제외한 특수 토큰과 본문 토큰 벡터를 모두 저장함.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 즉각적인 고정밀 검색 결과 확인 (우선순위: P1)

사용자가 Django 관련 기술 질문을 입력했을 때, 시스템은 1초 이내에 가장 관련성 높은 문서를 최상단에 배치하여 응답해야 합니다. 사용자는 검색 품질의 저하 없이 기존보다 훨씬 빠른 속도로 정보를 찾을 수 있습니다.

**우선순위 이유**: 검색 엔진의 핵심 가치는 '정확도'와 '속도'입니다. 기존 1.5초의 지연은 사용자 경험을 저해하므로 이를 1초 이내로 단축하여 쾌적한 환경을 제공하는 것이 목표입니다.

**독립적 테스트**: Playground UI 또는 MCP 도구를 통해 질문을 던지고, 전체 응답 시간이 1초 이내인지 확인하며, 반환된 결과의 상위 3개가 질문 의도와 일치하는지 검증합니다.

**수락 시나리오(Acceptance Scenarios)**:

1. **Given** 대량의 Django 문서 데이터가 인덱싱된 상태에서, **When** 사용자가 "How to filter querysets"와 같은 구체적인 질문을 입력하면, **Then** 1,000ms 이내에 검색 결과가 반환되어야 합니다.
2. **Given** 검색 결과가 반환될 때, **When** 상위 결과를 확인하면, **Then** Reranker(MaxSim)에 의해 재정렬된 가장 적합한 문서 청크가 1위에 위치해야 합니다.

---

### 사용자 스토리 2 - 하이브리드 검색의 효율적 관리 (우선순위: P2)

관리자 또는 개발자는 기존의 복잡한 SQL 쿼리 대신 Django ORM 스타일로 하이브리드 검색 로직을 유지보수할 수 있어야 합니다.

**우선순위 이유**: 코드의 가독성과 유지보수성은 장기적인 프로젝트 운영에 필수적입니다. ParadeDB 통합을 통해 이를 개선합니다.

**독립적 테스트**: `django-paradedb`를 활용한 쿼리셋 호출이 기존 Raw SQL과 동일하거나 더 나은 성능의 검색 후보군(Top-N)을 추출하는지 단위 테스트로 검증합니다.

**수락 시나리오**:

1. **Given** `DocumentChunk` 모델에 대해, **When** `django-paradedb` 전용 매니저를 통해 하이브리드 검색을 수행하면, **Then** BM25와 벡터 검색 결과가 결합된 쿼리셋이 반환되어야 합니다.

---

### 예외 케이스 (Edge Cases)

- **멀티 벡터 데이터 누락**: 특정 청크에 리랭킹용 멀티 벡터 데이터가 없는 경우, 시스템은 하이브리드 검색 결과(1차 점수)를 기반으로 Fallback 처리를 해야 합니다.
- **관련성 낮은 결과 처리**: 리랭킹 결과가 임계값(0.3) 미만인 경우, 사용자에게 검색 결과가 없음을 알리거나 적절한 안내 문구를 출력해야 합니다.

## 요구 사항 *(필수)*

### 기능적 요구 사항 (Functional Requirements)

- **FR-001**: 시스템은 `gpahal/bge-m3-onnx-int8` 모델을 사용하여 단 한 번의 추론으로 Dense 벡터와 멀티 벡터를 동시에 생성해야 함.
- **FR-002**: 시스템은 `django-paradedb`를 활용하여 BM25와 벡터 검색을 결합한 RRF(Reciprocal Rank Fusion) 기반 하이브리드 검색을 수행해야 함.
- **FR-003**: 하이브리드 검색을 통해 **상위 50개**의 1차 검색 후보군을 확보해야 함.
- **FR-004**: 시스템은 1차 검색 결과에 대해 Late Interaction(MaxSim) 연산을 수행하여 최종 순위를 결정해야 함. 리랭킹 연산 실패 시 하이브리드 검색 결과를 반환하는 폴백 로직을 포함함.
- **FR-005**: 리랭킹용 멀티 벡터는 문서 수집(Ingestion) 시점에 미리 계산되어 저장되어야 함.
- **FR-006**: 리랭킹 결과 반영 시 **최소 관련성 점수(Threshold) 0.3**을 적용하여 필터링해야 함. 임계값 미만 시 빈 결과를 반환함.

### 시스템 제약 사항 (System Constraints - Django Docs MCP)

- **SYS-001**: 모든 추론 연산은 CPU 환경의 ONNX Runtime에서 실행되어야 하며, 전체 검색 프로세스(Retrieval + Rerank)는 1,000ms 이내에 완료되어야 함.
- **SYS-002**: 검색 API 반환 포맷은 `search_django_knowledge` 규격(`content`, `target_version`, `document_type`, `source_url`, `relevance_score`, `extra_meta`)을 반드시 준수해야 함.
- **SYS-003**: 리랭킹용 멀티 벡터는 Postgres의 `bytea` 필드에 128차원/int8 양자화된 바이너리 형태로 저장되어야 함. 특수 토큰은 포함하되 패딩 토큰은 제외함(Pruning). 바이너리 시작 2바이트에 토큰 수를 기록함.

### 주요 엔티티

- **DocumentChunk**: 기존 청크 모델에 `multi_vector_low_dim` (압축된 멀티 벡터 데이터) 필드가 추가된 형태.
- **SearchProcessor**: 하이브리드 검색과 Late Interaction 리랭킹을 순차적으로 실행하는 서비스 레이어.

### 가정 (Assumptions)
- **AS-001**: 한 번 인제션된 문서 청크는 수정되지 않음(Immutable). 본문 수정 시 해당 청크는 삭제 후 재인제션됨을 전제로 함.

## 성공 기준 *(필수)*

### 측정 가능한 결과 (Measurable Outcomes)

- **SC-001**: 검색 질의 시작부터 최종 결과 반환까지의 평균 소요 시간이 **1,000ms 이하**여야 함.
- **SC-002**: 리랭킹을 거친 상위 1위 결과의 정확도가 기존 Cross-Encoder 방식 대비 유사하거나 개선되어야 함.
- **SC-003**: 1만 개 청크 기준, 추가되는 멀티 벡터 데이터의 저장 용량이 **1GB 이내**로 관리되어야 함.
