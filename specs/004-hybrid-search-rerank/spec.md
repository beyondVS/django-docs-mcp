# 기능 명세서: 하이브리드 검색 및 Rerank 전략을 통한 검색 품질 개선

**기능 브랜치**: `004-hybrid-search-rerank`
**생성일**: 2026년 3월 18일
**상태**: 초안(Draft)
**입력**: 사용자 설명: "청크 검색시의 검색 전략 보완. 하이브리드 검색(bm25+벡터) + rerank 전략으로 좀더 양질의 검색 결과를 리턴 한다."

## Clarifications

### Session 2026-03-18

- Q: 하이브리드 검색의 BM25(키워드) 구현 방식은? → A: PostgreSQL의 `pg_search` (ParadeDB) 확장을 사용하여 고성능 BM25 인덱싱을 수행한다.
- Q: Reranker 모델 선정 및 최적화 전략은? → A: 사전 양자화된 `tss-deposium/bge-reranker-v2-m3-onnx-int8` 모델을 선택하며, CPU 환경 최적화를 위해 임베딩 모델과 Reranker 모두 `ONNX Runtime`을 적용한다.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 용어 기반 검색 정확도 향상 (우선순위: P1)

사용자는 "Filter"와 같은 특정 Django 메서드나 기술 용어를 검색할 때, 의미론적으로 유사한 내용뿐만 아니라 해당 용어가 명시적으로 포함된 가장 관련성 높은 문서를 먼저 찾기를 원합니다.

**우선순위 이유**: 기술 문서 검색에서 핵심 용어의 일치는 사용자가 기대하는 가장 기본적인 검색 품질입니다.

**독립적 테스트**: 특정 클래스명이나 메서드명을 검색했을 때, 하이브리드 검색을 통해 해당 용어가 포함된 청크가 상위에 노출되는지 확인하여 테스트할 수 있습니다.

**수락 시나리오(Acceptance Scenarios)**:

1. **Given** "QuerySet filter"라는 검색어가 주어졌을 때, **When** 하이브리드 검색을 수행하면, **Then** 단순 벡터 검색보다 해당 키워드가 포함된 기술 참조 문서가 검색 결과의 상단에 배치되어야 합니다.
2. **Given** 검색 결과가 반환되었을 때, **When** relevance_score를 확인하면, **Then** 하이브리드 결합 방식에 따른 통합 점수가 제공되어야 합니다.

---

### 사용자 스토리 2 - Rerank를 통한 검색 결과 정교화 (우선순위: P1)

시스템은 1차적으로 검색된 다수의 후보 청크들 중에서, 사용자의 질문 의도와 가장 밀접하게 부합하는 상위 결과를 다시 선별하여 최상의 정보를 제공해야 합니다.

**우선순위 이유**: 1차 검색(Retrieval)은 재현율(Recall)에 집중하는 경향이 있으므로, 답변의 정확도를 높이기 위해 정밀도(Precision)를 보강하는 Rerank 과정이 필수적입니다.

**독립적 테스트**: 초기 검색 결과 리스트를 Reranker 모델에 통과시켰을 때, 정답에 더 가까운 문서의 순위가 상승하는지 검증하여 테스트할 수 있습니다.

**수락 시나리오**:

1. **Given** 1차 검색을 통해 20개의 후보 청크가 확보되었을 때, **When** Rerank 프로세스를 거치면, **Then** 사용자의 질문에 대한 '직접적인 해결책'을 담은 청크가 최상위(Top-1)로 재배치되어야 합니다.
2. **Given** Rerank를 거친 결과가 반환될 때, **When** relevance_score를 확인하면, **Then** Reranker 모델이 산출한 정교한 유사도 점수로 업데이트되어야 합니다.

---

### 사용자 스토리 3 - 검색 지연 시간 최소화 (우선순위: P2)

사용자는 검색 전략이 고도화되더라도 여전히 즉각적인 검색 결과를 얻기를 원합니다. 하이브리드 검색과 Rerank 단계가 추가됨에 따른 지연 시간이 체감되지 않아야 합니다.

**우선순위 이유**: 검색 품질만큼 중요한 것이 시스템의 응답성입니다.

**독립적 테스트**: 대량의 요청 상황에서 검색 및 Rerank를 포함한 전체 응답 시간을 측정하여 성공 기준 충족 여부를 확인합니다.

**수락 시나리오**:

1. **Given** 하이브리드 검색과 Rerank가 활성화된 상태에서, **When** 검색 요청을 보내면, **Then** 전체 응답 시간이 설정된 임계값 이내여야 합니다.

### 예외 케이스 (Edge Cases)

- **검색 결과가 없는 경우**: BM25와 벡터 검색 모두에서 결과를 찾지 못했을 때 시스템은 어떻게 반응합니까? (적절한 빈 결과 반환 및 에러 메시지 방지)
- **Reranker 모델 타임아웃**: Rerank 단계에서 지연이 발생하거나 모델 호출에 실패할 경우, 1차 검색 결과(Fallback)를 그대로 반환하는 정책이 필요합니다.
- **매우 짧은 쿼리**: "is", "the"와 같은 무의미한 키워드 쿼리에 대해 하이브리드 검색이 노이즈를 양산하지 않도록 처리해야 합니다.

## 요구 사항 *(필수)*

### 기능적 요구 사항 (Functional Requirements)

- **FR-001 (하이브리드 엔진)**: 시스템은 반드시 키워드 검색(BM25)과 벡터 유사도 검색(Embedding)을 동시에 수행하는 엔진을 갖추어야 함.
- **FR-002 (점수 결합 전략)**: 시스템은 **RRF (Reciprocal Rank Fusion)** 방식을 사용하여 BM25(키워드)와 벡터(유사도) 검색 순위를 결합한 통합 순위를 산출해야 함.
- **FR-003 (Rerank 단계)**: 시스템은 1차 검색(하이브리드) 결과 중 **상위 20개**의 항목을 대상으로 Reranker 모델을 사용하여 순위를 재정렬해야 함.
- **FR-004 (인터페이스 준수)**: 최종 검색 결과는 반드시 기존 `search_django_knowledge` 규격을 유지하며, `extra_meta`에 1차 검색 점수와 Rerank 점수를 포함할 수 있어야 함.
- **FR-005 (성능 평가 체계)**: 시스템은 검색 품질 개선을 객관적으로 측정하기 위해 **MRR(Mean Reciprocal Rank) 및 Hit Rate(Top-3)** 지표를 사용하며, 이를 위해 주요 질문 50개 이상의 **평가용 골든 데이터셋(Golden Dataset)을 신규 구축**하여 검증해야 함.

### 시스템 제약 사항 (System Constraints - Django Docs MCP)

- **SYS-001**: 모든 데이터 조회는 `pgvector` 기반의 인프라를 활용하되, BM25 키워드 검색을 위해 `pg_search` (ParadeDB) 확장을 도입하여 고성능 인덱싱 및 정밀한 점수 산출을 수행해야 함.
- **SYS-002**: 임베딩 모델(bge-m3)과 Reranker 모델이 검색 시점에 동시에 로드되므로 메모리 효율 및 CPU 추론 속도 최적화가 필수적임. 이를 위해 Reranker는 사전 양자화된 `tss-deposium/bge-reranker-v2-m3-onnx-int8` 모델을 사용하며, 두 모델 모두 `ONNX Runtime`을 적용하여 실행해야 함.
- **SYS-003**: 전체 검색 프로세스(Retrieval + Rerank)는 헌법의 실용주의 원칙에 따라 과도한 복잡성을 지양해야 함.

### 주요 엔티티

- **HybridRetriever**: BM25와 벡터 검색을 병렬로 수행하고 결과를 통합하는 객체.
- **Reranker**: 1차 결과를 재평가하여 순위를 보정하는 컴포넌트.
- **SearchEvaluationSet**: 검색 품질을 검증하기 위한 질문-정답 문서 쌍의 집합.

## 성공 기준 *(필수)*

### 측정 가능한 결과 (Measurable Outcomes)

- **SC-001**: 기존 단일 벡터 검색 대비, 주요 기술 용어 검색에 대한 상위 3개 결과(Top-3)의 정답 포함 비율(Hit Rate)이 20% 이상 향상됨.
- **SC-002**: Rerank를 거친 검색 결과의 평균 역순위 점수(MRR)가 기존 대비 0.15 이상 개선됨.
- **SC-003**: 사용자의 질의 의도와 관련 없는 '허위 양성(False Positive)' 결과가 검색 결과 상위 5개 내에서 1건 이하로 감소함.
