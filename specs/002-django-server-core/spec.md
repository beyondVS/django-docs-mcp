## Clarifications

### Session 2026-03-16
- Q: 동일한 소스 경로(URL/Path)를 가진 문서가 이미 존재할 때 시스템은 어떻게 동작해야 합니까? → A: 기존 문서와 연관된 모든 Chunk를 삭제한 후 새롭게 다시 적재(Upsert)함.
- Q: 시스템은 동일한 문서의 서로 다른 버전을 어떻게 관리하고 구분해야 합니까? → A: 문서(Document) 엔티티에 version 필드를 추가하여 버전별로 독립된 엔티티로 취급하고 검색 시 필터링을 지원함.
- Q: Playground 페이지와 검색 기능을 사용할 수 있는 대상은 누구입니까? → A: 로그인을 완료한 모든 내부 사용자(Staff)가 사용 가능함.
- Q: 대량 적재 작업 중 일부 문서에서 오류가 발생할 경우 시스템은 어떻게 처리해야 합니까? → A: 개별 문서 단위로 트랜잭션을 처리하여 성공한 항목은 커밋하고 실패한 항목은 별도 로그에 기록함.
- Q: Playground 웹 인터페이스에서 검색 질의 외에 추가적으로 제공해야 할 필터링 조건은 무엇입니까? → A: Django 버전과 문서 카테고리(유형) 필터링 기능을 제공함.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 문서 적재 자동화 (우선순위: P1)

관리자는 로컬에 수집된 마크다운 문서들을 시스템에 일괄 적재할 수 있어야 합니다. 시스템은 문서의 메타데이터를 파싱하고, 검색에 최적화된 단위(Chunk)로 분할하여 벡터 데이터베이스에 저장합니다.

**우선순위 이유**: 검색 시스템의 핵심인 데이터를 공급하는 가장 기초적이고 필수적인 단계입니다.

**독립적 테스트**: `python manage.py ingest_docs`와 같은 명령어를 통해 로컬 마크다운 파일을 읽고, 데이터베이스에 정상적으로 문서-섹션-조각 계층 구조가 생성되는지 확인하여 테스트할 수 있습니다.

**수락 시나리오(Acceptance Scenarios)**:

1. **Given** `data_sources/` 디렉토리에 마크다운 파일이 존재할 때, **When** 적재 명령을 실행하면, **Then** 각 문서의 YAML Front Matter가 파싱되어 데이터베이스에 저장됨.
2. **Given** 긴 문장으로 구성된 문서가 있을 때, **When** 적재 프로세스가 진행되면, **Then** 500~800 토큰 범위 내에서 10% 오버랩을 유지하며 조각(Chunk)으로 분할됨.
3. **Given** 문서 내에 파이썬 코드 블록이 포함되어 있을 때, **When** 청킹(Chunking)이 수행되면, **Then** 코드 블록은 중간에 잘리지 않고 하나의 조각에 보존됨.

### 사용자 스토리 2 - 검색 실험실 (Playground) (우선순위: P1)

사용자는 웹 인터페이스를 통해 자연어로 질문을 입력하고, 시스템에 저장된 데이터 중 유사도가 높은 결과들을 실시간으로 확인할 수 있습니다. 사용자는 특정 Django 버전이나 문서 카테고리를 선택하여 검색 범위를 제한할 수 있습니다.

**우선순위 이유**: 적재된 데이터의 검색 품질을 직접 확인하고 튜닝하기 위한 필수 인터페이스입니다.

**독립적 테스트**: 웹 브라우저에서 Playground 페이지에 접속하여 검색어를 입력하고, 필터를 적용했을 때 반환된 결과가 해당 버전/카테고리에 속하는지 확인하여 테스트할 수 있습니다.

**수락 시나리오**:

1. **Given** Playground 검색창에 "QuerySet"을 입력하고 '버전: 5.0' 필터를 선택하면, **Then** 5.0 버전 문서에서 추출된 조각들만 검색 결과에 표시됨.
2. **Given** 검색 결과가 반환될 때, **Then** 각 결과 항목에 원본 문서의 URL, 버전, 섹션 제목이 함께 표시됨.

### 사용자 스토리 3 - 데이터 및 품질 관리 (우선순위: P2)

관리자는 Django Admin 인터페이스를 통해 적재된 문서의 상태(활성/비활성)를 관리하고, 검색 결과와 원본 본문을 대조하여 품질을 분석할 수 있습니다.

**우선순위 이유**: 운영 환경에서 데이터의 유효성을 제어하고 검색 성능을 지속적으로 개선하기 위해 필요합니다.

**독립적 테스트**: Django Admin에 로그인하여 특정 문서의 메타데이터를 수정하거나 삭제하고, 이것이 실제 검색 결과에 반영되는지 확인하여 테스트할 수 있습니다.

**수락 시나리오**:

1. **Given** 특정 문서를 '비활성' 상태로 변경하면, **Then** 이후 Playground 검색 결과에서 해당 문서의 조각들이 제외됨.
2. **Given** 품질 분석 화면에서, **Then** 유사도 점수와 함께 원본 마크다운 본문을 즉시 대조하여 확인할 수 있음.

### 예외 케이스 (Edge Cases)

- **중복 문서 처리**: 동일한 경로의 문서가 이미 적재되어 있는 경우, 기존 문서를 삭제하고 새로운 내용으로 교체(Upsert)하여 항상 최신 상태를 유지하고 데이터 중복을 방지함.
- **적재 중 일부 문서 실패**: 대량 적재 시 개별 문서 단위로 트랜잭션을 처리하며, 실패한 문서는 DB에 반영하지 않고 오류 로그를 생성하여 관리자가 확인할 수 있도록 함.
- **임베딩 모델 오류**: 외부/로컬 임베딩 모델 서버가 응답하지 않을 때, 적재 프로세스가 안전하게 중단되고 상태가 로깅되어야 합니다.
- **매우 짧은 문서**: 500토큰 미만의 짧은 문서는 분할 없이 하나의 조각으로 처리되어야 합니다.

## 요구 사항 *(필수)*

### 기능적 요구 사항 (Functional Requirements)

- **FR-001**: 시스템은 반드시 로컬 파일 시스템의 마크다운 문서를 읽고 YAML Front Matter(URL, 작성일, 태그 등)를 추출해야 함.
- **FR-002**: 시스템은 문서를 500~800 토큰 단위로 분할하되, 10% 오버랩 유지 및 검색 품질 향상을 위해 생성된 **각 조각 상단에 문서 제목과 원본 URL을 주입(문맥 보강)**해야 함.
- **FR-003**: 시스템은 파이썬 코드 블록(` ```python `)이 청크 중간에 잘리지 않도록 보호 로직을 적용해야 함.
- **FR-004**: 시스템은 BAAI/bge-m3 모델을 사용하여 각 조각의 벡터 표현(1024차원)을 생성해야 함.
- **FR-005**: 시스템은 PostgreSQL의 `pgvector` 확장을 사용하여 조각의 벡터 및 본문, 메타데이터를 저장해야 함.
- **FR-006**: 시스템은 사용자에게 자연어 질의를 입력받아 코사인 유사도 기반의 검색 결과를 반환하며, 대상 버전(`target_version`) 및 카테고리별 필터링이 가능한 웹 UI(Playground)를 제공해야 함.
- **FR-007**: 시스템은 관리자 권한이 있는 사용자만 데이터 적재 및 설정을 변경할 수 있도록 제어해야 하며, Playground는 인증된 모든 Staff 사용자가 접근 가능해야 함.

### 시스템 제약 사항 (System Constraints - Django Docs MCP)

- **SYS-001**: 데이터 구조는 문서(Document) - 섹션(Section) - 조각(Chunk)의 3단계 계층 구조를 반드시 유지해야 함.
- **SYS-002**: 전체 시스템은 Docker 및 Docker Compose를 통해 컨테이너화되어 구동 가능해야 함.
- **SYS-003**: `mcp_server`와 동일한 PostgreSQL 데이터베이스 인스턴스를 공유하여 사용할 수 있어야 함.
- **SYS-004**: 크롤러 및 MCP 서버와는 표준화된 데이터 규격(ID, URL, Content, Vector)을 통해 독립적으로 동작해야 함.

### 주요 엔티티

- **Document**: 원본 문서 정보 (제목, 버전, 카테고리, 원본 경로, 소스 유형 등).
- **Section**: 문서 내 논리적 구분 (헤더 기반 섹션).
- **Chunk**: 검색의 최소 단위. 본문 내용, 벡터 데이터, 원본 위치 정보를 포함함.

## 성공 기준 *(필수)*

### 측정 가능한 결과 (Measurable Outcomes)

- **SC-001**: 100개의 마크다운 문서를 적재할 때, 데이터 누락 없이 모든 문서가 5분 이내에 데이터베이스에 적재되어야 함.
- **SC-002**: Playground에서 검색을 수행할 때, 1초 이내에 가장 유사한 상위 5개의 결과가 유사도 점수와 함께 반환되어야 함.
- **SC-003**: 적재된 데이터 중 무작위로 추출한 10개의 파이썬 코드 블록이 청킹 후에도 문법적 파손 없이 온전히 보존되어야 함.
- **SC-004**: 시스템 구성 요소를 Docker Compose로 실행했을 때, 추가 설정 없이 데이터베이스 연결 및 서비스 구동이 성공해야 함.
