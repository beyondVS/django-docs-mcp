## Clarifications

### Session 2026-03-15
- Q: 문서 업데이트 전략 → A: 전체 재수집 (Full Crawl) - 실행 시마다 전체 내용을 새로 수집하여 최신 상태를 보장함.
- Q: 이미지 데이터 처리 전략 → A: URL 보존 - 이미지 태그를 마크다운 형식으로 남기되 파일은 다운로드하지 않아 인덱싱 노이즈를 방지함.
- Q: 로컬 파일 저장 구조 → A: 계층 구조 유지 - 원본 웹사이트의 URL 경로에 맞춰 하위 디렉토리를 생성하여 저장함.
- Q: 본문 추출 기술 전략 → A: Readability 알고리즘 - 휴리스틱 기반으로 본문 영역을 자동으로 감지하여 유지보수성을 높임.
- Q: 메타데이터 기록 방식 → A: YAML Front Matter - 마크다운 파일 상단에 메타데이터 블록을 삽입하여 데이터 정형성을 확보함.
- Q: 정적 파일(Assets) 저장 범위 → A: 텍스트 전용 수집 - 이미지 등 미디어 파일은 저장하지 않고 마크다운 텍스트와 원본 URL 링크만 보존함.
- Q: 크롤링 탐색 범위 → A: 내부 도메인 한정 - 시작 URL의 하위 경로 및 동일 도메인 내 내부 링크만 재귀적으로 수집하여 외부 유출을 방지함.
- Q: 동적 렌더링 지원 여부 → A: 정적 파싱 - JavaScript 실행 없이 초기 HTML만 수집하여 효율성과 속도를 최적화함.
- Q: 크롤링 동시성 및 Rate Limit 대응 → A: 제한적 동시성(최대 3~5개)을 기본으로 하되, 429(Too Many Requests) 오류 발생 시 즉시 대기 시간을 늘리는 동적 적응형 지연 전략을 사용함.
- Q: 진행 상태 관리 및 재시도 전략 → A: 메모리 기반 재시도 - 실패한 URL을 최대 3회까지 지수 백오프 방식으로 재시도하며, 별도의 체크포인트 파일은 생성하지 않음.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - Django ORM Cookbook 문서의 자동 크롤링 및 로컬 저장 (우선순위: P1)

사용자는 지정된 Django ORM Cookbook URL에서 모든 챕터의 내용을 자동으로 수집하여 로컬에 저장하고자 합니다.

**우선순위 이유**: RAG 시스템 구축을 위한 가장 기초적인 단계로, 데이터가 수집되어야 이후 임베딩 및 검색이 가능합니다.

**독립적 테스트**: `data_sources/` 디렉토리에 Cookbook별로 구조화된 문서 파일들이 생성되는지 확인함으로써 테스트할 수 있습니다.

**수락 시나리오(Acceptance Scenarios)**:

1. **Given** Django ORM Cookbook URL이 제공되었을 때, **When** 크롤러를 실행하면, **Then** 정적 HTML 파싱 및 내부 도메인 한정 탐색을 통해 모든 챕터 내용이 누락 없이 수집되어야 함.
2. **Given** 수집된 HTML 데이터가 있을 때, **When** 저장 프로세스가 진행되면, **Then** 웹사이트의 계층 구조를 반영한 디렉토리에 파일이 생성되어야 함.

---

### 사용자 스토리 2 - 수집된 문서의 마크다운 변환 및 전처리 (우선순위: P1)

사용자는 수집된 HTML 문서를 RAG 최적화 및 가독성을 위해 마크다운 형식으로 변환하여 보관하고자 합니다.

**우선순위 이유**: HTML 태그는 임베딩 시 노이즈가 될 수 있으므로, 의미 중심의 마크다운 변환은 검색 품질 향상에 필수적입니다.

**독립적 테스트**: 생성된 마크다운 파일이 원본 HTML의 구조(제목, 코드 블록, 목록 등)를 올바르게 유지하고 있는지 확인함으로써 테스트할 수 있습니다.

**수락 시나리오**:

1. **Given** 수집된 HTML 문서가 있을 때, **When** 변환 도구를 실행하면, **Then** Readability 알고리즘을 통해 추출된 본문과 링크, 이미지(URL 보존), 특히 Python 코드 블록이 보존된 마크다운 파일이 생성되어야 함.

---

### 사용자 스토리 3 - 문서 버전 및 메타데이터 관리 (우선순위: P2)

사용자는 수집된 문서가 Django 2.x 기반임을 명시하고, 이를 RAG 시스템에서 구분하여 검색할 수 있도록 메타데이터를 부여하고자 합니다.

**우선순위 이유**: 오래된 버전의 지식이 최신 버전과 충돌할 수 있으므로, 버전 정보를 명확히 관리하는 것이 RAG의 답변 정확도에 중요합니다.

**독립적 테스트**: 저장된 각 문서 파일의 상단(YAML Front Matter 등)에 `target_version: 2.x` 정보가 포함되어 있는지 확인함으로써 테스트할 수 있습니다.

**수락 시나리오**:

1. **Given** 문서를 저장할 때, **When** 메타데이터 부여 옵션을 활성화하면, **Then** 해당 문서 상단에 YAML Front Matter 형식으로 원본 URL과 대상 Django 버전 정보가 기록되어야 함.

---

### 예외 케이스 (Edge Cases)

- **네트워크 오류**: 크롤링 중 특정 페이지 접근이 실패할 경우, 시스템은 최대 3회까지 지수 백오프 기반으로 재시도해야 함.
- **Rate Limit(429) 발생**: 시스템은 즉시 요청을 중단하고 대기 시간을 동적으로 늘린 후 재시도해야 함.
- **중복 문서**: 동일한 내용의 문서가 다른 경로로 존재할 경우, 해시 기반의 중복 제거 전략을 적용해야 함.
- **비표준 HTML 구조**: Cookbook의 특정 페이지가 표준 레이아웃과 다를 경우, 본문 영역을 정확히 추출하기 위한 폴백(Fallback) 추출 전략이 필요함.

## 요구 사항 *(필수)*

### 기능적 요구 사항 (Functional Requirements)

- **FR-001**: 시스템은 Django ORM Cookbook의 시작 URL로부터 내부 도메인 내 링크만 재귀적으로 탐색하여 수집해야 함.
- **FR-002**: 수집된 HTML은 Readability 알고리즘을 사용하여 본문 영역을 자동으로 감지하고, 내비게이션 등 불필요한 요소를 제외한 후 마크다운으로 변환해야 함.
- **FR-003**: [합리적 기본값] 저장 경로는 프로젝트의 명확성을 위해 다음과 같이 구조화하며, 원본 URL 구조에 따른 계층적 디렉토리를 생성함:
  - `data_sources/django2-orm-cookbook/`
- **FR-004**: 모든 변환된 마크다운 파일 상단에는 YAML Front Matter 형식으로 `target_version: 2.x`, `source_url`, 그리고 수집 시점(`collected_at`) 메타데이터를 포함해야 함.
- **FR-005**: 크롤링 시 정적 HTML 파싱 방식을 사용하며, JavaScript 실행은 배제함.
- **FR-006**: 시스템은 최대 3~5개의 동시 요청을 수행하되, Rate Limit 오류 발생 시 적응형 대기 전략을 적용함.
- **FR-007**: 마크다운 변환 시 이미지는 다운로드하지 않고 원본 URL 링크(`![alt](url)`)만 유지해야 함.
- **FR-008**: 실패한 URL에 대해 메모리 기반으로 최대 3회 지수 백오프 재시도를 수행함.

### 시스템 제약 사항 (System Constraints - Django Docs MCP)

- **SYS-001**: 데이터 수집 범위는 크롤링 및 마크다운 생성에 한정하며, 청킹 및 벡터 DB 구축은 본 기능의 범위에서 제외함.
- **SYS-002**: 변환된 마크다운은 추후 RAG 시스템에서 활용 가능하도록 구조적 정밀도(헤딩 계층, 코드 블록)를 유지해야 함.
- **SYS-003**: 수집된 모든 파일은 UTF-8 인코딩으로 저장되어야 함.

### 주요 엔티티 (데이터 관련 기능인 경우 포함)

- **DataSource**: 크롤링 대상 URL, 이름, 대상 버전 정보를 포함하는 수집 단위.
- **DocumentFile**: 마크다운으로 변환된 개별 문서 파일 및 관련 메타데이터.

## 성공 기준 *(필수)*

### 측정 가능한 결과 (Measurable Outcomes)

- **SC-001**: 지정된 Django ORM Cookbook의 모든 챕터(약 100% 수집율)가 로컬 마크다운 파일로 변환되어 저장됨.
- **SC-002**: 변환된 마크다운 파일의 95% 이상이 깨진 링크나 손상된 코드 블록 없이 원본의 내용을 충실히 반영함. (무작위 샘플링 검수 및 T019 최종 검증 단계에서 확인)
- **SC-003**: 모든 수집된 문서 상단에 `target_version: 2.x` 메타데이터가 정확히 태깅되어 검색 필터링이 가능함.
- **SC-004**: 수집 및 변환 프로세스가 자동화되어, 단일 명령 실행으로 전체 워크플로우가 완료됨.
