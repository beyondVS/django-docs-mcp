# 임베딩 및 리트리벌 전략서 (Retrieval & Ranking Spec)

> 💡 **핵심 목표**
> 텍스트 문서를 벡터화하여 데이터베이스에 효율적으로 저장하고,
> 질문과 가장 관련성이 높은 최적의 문서를 빠르고 정확하게 찾아내기 위한
> **하이브리드 리트리벌 및 Reranking 전략**을 정의합니다.

---

## 1. 모델 및 런타임 (Models & Runtime)

인프라 비용을 낮추고 속도를 극대화하기 위해, CPU 환경에 최적화된 경량/양자화 모델을 활용합니다.

| 용도 | 모델명 | 주요 특징 및 최적화 전략 |
| :--- | :--- | :--- |
| **Embedding** | `BAAI/bge-m3` | 다국어 검색 특화, 1024 차원 단일 벡터 생성, 8,192 토큰 컨텍스트 지원.<br>**ONNX Runtime 가속**을 통해 CPU 지연 시간 최소화. |
| **Reranker** | `keisuke-miyako/`<br>`bge-reranker-base-onnx-int8` | 질문과 문서 쌍의 의미적 연관성을 0~1 사이로 정밀 점수화.<br>**INT8 양자화 ONNX 모델** 적용으로 메모리 절감 및 실행 속도 극대화. |

## 2. 청킹 전략 (Chunking Strategy)

문서의 의미론적 구조를 보존하기 위해 2단계(2-Phase) 청킹 파이프라인을 수행하며, 모델의 한계를 고려해 파라미터를 엄격하게 제어합니다.

*   **Phase 1: 논리적 분할 (`MarkdownHeaderTextSplitter`)**
    마크다운의 헤더(H1~H3)를 기준으로 문서를 논리적 섹션으로 1차 분할합니다.
*   **Phase 2: 물리적 재분할 (`MarkdownTextSplitter`)**
    기술 문서의 핵심인 **코드 블록(```) 보호를 최우선**으로 하여 텍스트를 크기 단위로 2차 분할합니다.

> ⚠️ **핵심 파라미터 선정 근거**
> *   **Chunk Size (1,500자):** 리랭커 모델의 **가장 엄격한 제약인 512 토큰 제한**을 고려한 수치입니다. 사용자 질문과 중첩(Overlap) 구간을 제외하더라도 문서의 핵심 본문이 강제로 절단(Truncation)되지 않도록 여유 있게 설정했습니다. (한글/영문 혼용 기준 약 600~800 토큰 내외 해당)
> *   **Overlap (200자):** 분할된 청크 간의 문맥 연속성을 유지하고, 문장 중간이 잘리는 지점에서의 치명적인 정보 손실을 방지하는 최소한의 안전 장치입니다.

## 3. 리트리벌 및 순위화 전략 (Retrieval & Ranking)

단순 벡터 검색이 가지는 한계(고유명사 누락 등)를 극복하기 위해, 3단계로 이어지는 촘촘한 파이프라인을 가동합니다.

### Step 1. 하이브리드 리트리벌 (Hybrid Retrieval)
*   **Vector Search:** `pgvector`를 활용한 코사인 유사도 검색으로 '문맥과 의미'가 유사한 문서를 포착합니다.
*   **Keyword Search:** `pg_search` (BM25)를 활용한 키워드 매칭으로 '고유 명사, API 명칭, 에러 코드' 등의 정확도를 강제로 확보합니다.

### Step 2. RRF 통합 (Reciprocal Rank Fusion)
벡터 점수와 키워드 점수는 스케일이 다르기 때문에 직접 합산할 수 없습니다. 따라서 순위 정보만을 활용하는 공식을 사용해 상위 20개 후보를 도출합니다.

*   **통합 공식:** $Score = \sum \frac{1}{rank + k}$ (k=60 기본값 적용)
*   **설계 근거:** 점수 스케일의 편향에 휘둘리지 않고 두 검색 결과를 가장 공평하고 안정적으로 결합하는 업계 표준 방식입니다. 상수 `k=60`은 하위 순위의 노이즈를 배제하면서 상위 문서의 변별력을 높이는 최적의 값입니다.
*   **구현 효율성:** 데이터베이스 내부(PostgreSQL)에서 `WITH` 절과 `FULL OUTER JOIN`을 사용해 SQL 단에서 결합 연산을 완료한 후, 최종 후보군(Top-20) 텍스트만 애플리케이션 계층으로 가볍게 전송합니다.

### Step 3. ONNX Reranking (최종 정밀도 확보)
*   **스코어 재산출:** RRF로 추출된 1차 상위 20개 청크를 Reranker 모델에 입력하여, 질문과 청크의 1:1 문맥적 연관성 점수(0~1)를 정밀하게 재산출합니다.
*   **임계값 필터링:** 계산 결과 관련성이 현저히 낮다고 판단되는 청크(예: 점수 0.1 이하)는 최종 결과에서 과감히 제외하여 LLM의 환각(Hallucination) 가능성을 차단합니다.
*   **성능 최적화:** 리랭킹은 매우 무거운 CPU 연산이므로 `sync_to_async` 래퍼를 사용하여 메인 비동기 이벤트 루프의 차단을 방지합니다.

## 4. 데이터 계층화 및 문맥 보존 (Context Preservation)

조각난 청크가 본래의 문맥을 잃지 않도록 메타데이터를 검색에 적극 활용합니다.

*   **가상 컬럼 결합 검색:** BM25 검색을 수행할 때 순수 본문(`content`)만 스캔하지 않습니다. 상위 섹션의 제목(`section_title`)을 가상 컬럼으로 덧붙여 검색함으로써, 본문만으로는 유추하기 힘든 문맥 정보를 보완하고 검색 정확도를 크게 끌어올립니다.
*   **코드 블록 무결성 보호:** 청킹 과정에서 코드 블록(```)이 중간에 절단될 위기에 처하면, 무조건 해당 코드 블록 전체를 하나의 청크에 통째로 포함시키는 **'문맥 우선' 예외 전략**을 취합니다.

---

## 5. 품질 평가 및 유지보수 (Quality Assurance)

지속적인 검색 품질 향상을 위한 모니터링 체계입니다.

*   **평가 지표 (Metrics):** 시스템의 검색 성능은 주로 MRR (Mean Reciprocal Rank)과 Hit Rate @5 지표를 기준으로 주기적으로 측정됩니다.
*   **골든 데이터셋 (Golden Dataset):** 기술 문서에서 자주 묻는 핵심 질문 50여 개와 그에 해당하는 완벽한 정답 문서 ID를 매칭해 둔 평가셋입니다. 청킹 파라미터나 검색 로직을 변경할 때, 성능이 후퇴(Regression)하지 않았는지 검증하는 기준점이 됩니다.
*   **Playground 피드백 튜닝:** 관리자 UI(Playground)에서 시각화된 하이브리드 점수와 리랭킹 점수 분포를 실시간으로 분석하여, 특정 키워드에 대한 BM25 가중치 등을 상황에 맞게 미세 조정합니다.
