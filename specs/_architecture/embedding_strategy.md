# 데이터 청킹 및 임베딩 전략서 (Chunking & Embedding Spec)

문서를 벡터화하여 데이터베이스에 저장하기 위한 전략과 규칙을 정의하며, 현재 `django_server`에 구현된 방식입니다.

## 1. 임베딩 모델

*   **모델 명:** BAAI/bge-m3
*   **특징:** 다국어 처리에 특화되어 있으며(한국어 질의 -> 영어 문서 검색 성능 탁월), 1024 차원의 벡터를 생성합니다.
*   **인프라 가속**: GPU(CUDA)가 사용 가능한 경우 자동으로 감지하여 가속을 활용합니다.

## 2. 청킹 및 문맥 유지 전략

단순 글자 수가 아닌, 의미가 유지되는 계층 구조 기반의 2단계 청킹 파이프라인(Chaining)을 수행합니다.

### 2.1 2단계 파이프라인 (LangChain Text Splitters)
*   **Phase 1: MarkdownHeaderTextSplitter**: 마크다운 헤더(H1, H2, H3)를 기준으로 문서를 논리 섹션으로 1차 분할합니다. 추출된 헤더 정보는 메타데이터(`Header 1~3`)에 저장됩니다.
*   **Phase 2: MarkdownTextSplitter**: 1단계에서 분할된 섹션을 임베딩 모델에 적합한 크기로 재분할합니다. 이때 마크다운 문법을 인식하여 코드 블록(```) 내부가 잘리는 것을 최대한 방지합니다.

### 2.2 청크 크기 및 오버랩 (bge-m3 최적화)
*   **Target Size**: 개별 청크의 크기는 **1,500 ~ 2,500 문자(Character)** 내외로 유지합니다. bge-m3의 넓은 컨텍스트 윈도우(8,192 토큰)를 활용하여 코드 블록의 문맥을 온전히 보존합니다.
*   **Overlap**: 청크 간 문맥 단절을 방지하기 위해 약 200자 내외의 중첩(Overlap) 구간을 설정하되, 코드 블록 기호가 깨지지 않도록 구분자 우선순위를 조정합니다.

### 2.3 데이터 순수성 및 메타데이터 관리
*   **순수 텍스트 유지**: 청크 본문(`content`)에는 `Context:`나 `Document:`와 같은 외부 정보를 직접 주입하지 않습니다. 이는 임베딩 모델의 노이즈를 줄이고 데이터의 원본성을 유지하기 위함입니다.
*   **메타데이터 활용**: 문서 제목, Django 버전, 카테고리, 계층적 헤더 경로(`header_context`) 등 모든 문맥 정보는 `Document.metadata` 필드를 통해서만 관리하며, 검색 결과 반환 시 에이전트에게 함께 전달됩니다.

## 3. 소스별 전략 및 처리 프로세스

*   **파일 파싱**: `python-frontmatter`를 사용하여 YAML Front Matter(버전, 카테고리 등)를 추출합니다.
*   **Upsert 로직**: `source_path`와 `target_version`을 기준으로 기존 적재 내역이 있으면 삭제 후 재생성함으로써 데이터 중복을 방지합니다.
*   **트랜잭션 관리**: 개별 파일 단위로 `transaction.atomic`을 적용하여, 적재 중 에러 발생 시 데이터 정합성을 보장(Rollback)합니다.
