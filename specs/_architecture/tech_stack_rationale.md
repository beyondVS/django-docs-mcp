# 기술 스택 선정 근거서 (Technology Stack Rationale)

> 💡 **기술 선정 원칙**
> 본 문서는 Django Docs MCP 프로젝트에서 채택한 주요 기술 및 라이브러리의
> 선정 근거와 전환 이력을 상세히 기술합니다.
> 모든 기술 선택은 프로젝트 헌법에 명시된 **'실용주의'**와 **'책임 분리'** 원칙을 바탕으로 결정되었습니다.

---

## 1. 인프라 및 데이터베이스 (Infrastructure & DB)

### 1.1 데이터베이스 기술 전환 이력 (ParadeDB)

| 기존 기술 | 변경된 기술 | 기술 전환의 이유 및 선정 근거 |
| :--- | :--- | :--- |
| `pgvector/pgvector:pg16`<br>(단일 확장 이미지) | **`paradedb/paradedb`**<br>**(공식 통합 이미지)** | 초기에는 단순 벡터 검색용 이미지를 사용했으나, 하이브리드 검색 구현을 위해 **BM25 엔진(`pg_search`)이 사전 설치된 ParadeDB 이미지**가 관리 효율성과 확장성 면에서 압도적으로 우수하다고 판단했습니다. |
| 단순 Vector Search | **Hybrid Search**<br>**(Vector + BM25)** | 벡터 검색의 의미론적 유사성만으로는 Django 고유의 API 명칭(`filter()`, `QuerySet` 등)에 대한 **'키워드 정밀도'**를 확보하는 데 뚜렷한 한계가 있었습니다. |

*   **`pg_search`의 장점:** BM25를 데이터베이스 내부에 고성능 Rust 언어로 구현하여 정밀도가 매우 높습니다.
    또한 SQL 쿼리 레벨에서 RRF 통합을 처리하므로 어플리케이션 계층으로 데이터를 가져오는 네트워크 지연 시간을 최소화합니다.
*   **DB 드라이버 (`psycopg3`):** Django 5.x의 최신 비동기 기능을 병목 없이 최대로 활용하기 위해,
    `async/await`를 기본으로 지원하는 `psycopg 3.2.x` 버전을 채택했습니다.

### 1.2 Docker & Docker Compose
*   **선정 근거:** 프로젝트의 **'컨테이너 우선'** 원칙에 따라 개발 환경과 운영 환경의 100% 일치성을 보장합니다.
    특히 ParadeDB와 같이 복잡한 확장이 포함된 데이터베이스를 호스트 OS에 번거로운 설정 없이 즉시 실행할 수 있게 해줍니다.

## 2. 모델 런타임 및 최적화 (Model Runtime & Optimization)

### 2.1 추론 엔진 기술 전환 (ONNX Runtime)

| 기존 기술 | 변경된 기술 | 기술 전환의 이유 및 선정 근거 |
| :--- | :--- | :--- |
| `sentence-transformers`<br>(PyTorch 기반) | **`optimum`**<br>**(ONNX Runtime)** | 무거운 PyTorch 기반에서는 임베딩과 리랭커 모델이 각각 거대한 메모리를 점유하고, GPU가 없는 순수 CPU 환경에서 추론 속도가 심각하게 저하되는 문제를 해결하기 위해 전면 도입했습니다. |

*   **메모리 최적화:** 두 모델이 동일한 가벼운 `onnxruntime` 라이브러리를 공유하게 되어 컨테이너의 상주 메모리를 대폭 절감합니다.
*   **추론 가속 (INT8):** 사전에 양자화된(Quantized) INT8 모델을 선택하여 구동함으로써,
    CPU 환경임에도 불구하고 기존 FP32 모델 대비 약 2~3배 빠른 추론 속도를 달성했습니다.
*   **런타임 경량화:** PyTorch 전체 라이브러리 대신 추론 전용 런타임만을 사용하여 컨테이너 크기를 최적화했습니다.

## 3. 검색 및 리랭킹 AI 모델 (Search & Reranking Models)

### 3.1 Embedding Model: `BAAI/bge-m3`
*   **선정 근거:** **8,192 토큰**이라는 매우 넓은 컨텍스트 윈도우(Context Window)를 지원합니다.
    이는 기술 문서의 특성상 필수적인 '긴 파이썬 예제 코드 블록'을 중간에 강제로 절단하지 않고
    하나의 온전한 벡터로 임베딩할 수 있게 해줍니다.
    또한 다국어(Multilingual) 처리 성능이 세계 최고 수준이라, 한국어 기술 질문에 대해서도 높은 의미적 유사도를 보장합니다.

### 3.2 Reranker Model: `keisuke-miyako/bge-reranker-base-onnx-int8`
*   **선정 근거:** 110M 파라미터 수준의 Base 크기 모델은 CPU 환경에서 무거운 v2-m3(Large) 모델 대비
    압도적으로 빠른 실시간 추론 속도를 제공합니다.
    속도가 빠름에도 불구하고, 1차 하이브리드 검색으로 추출된 Top-20 후보군 사이의 순위를
    정밀하게 교정(Refinement)하는 데는 충분히 훌륭한 성능을 보여줍니다.

## 4. 백엔드 및 서빙 계층 (Backend & Serving)

### 4.1 수집 및 파싱 (`httpx`, `readability`, `markdownify`)
*   **선정 근거:** `httpx`의 비동기 I/O 처리와 `tenacity` 라이브러리의 지수 백오프(Exponential Backoff)를 결합하여 웹 크롤링 시의 수집 안정성을 극대화했습니다.
    또한 `readability`를 활용해 순수 본문만을 식별하고, `markdownify`로 코드를 마크다운으로 변환하여 벡터화 품질을 높였습니다.

### 4.2 코어 엔진 (Django)
*   **선정 근거:**
    *   **강력한 관리 도구(Admin):** 수집된 방대한 지식 문서와 벡터 데이터를 시각적으로 관리하고,
        Playground를 통해 검색 품질을 즉각적으로 테스트/튜닝할 수 있는 독보적인 관리자 UI를 기본 제공합니다.
    *   **안정적인 생태계:** LangChain 등 최신 RAG 생태계 라이브러리와의 연동이 용이하며,
        대규모 데이터 관리를 위해 오랜 기간 검증된 강력한 ORM과 DB 마이그레이션 시스템을 제공합니다.

### 4.3 에이전트 인터페이스 (FastMCP)
*   **선정 근거:** 복잡한 표준 MCP SDK보다 구조가 훨씬 가볍고 현대적인 비동기(Async) 문법을 지원합니다.
    Django가 처리한 검색 로직을 지연 없이 깔끔한 인터페이스로 AI 에이전트에게 노출하는 데 최적의 도구입니다.

## 5. 의존성 및 코드 품질 도구 (Tooling)

### 5.1 워크스페이스 관리자 (`uv`)
*   **선정 근거:** Rust 기반으로 제작되어 기존 `pip`이나 `poetry` 대비 압도적인 패키지 설치 속도를 자랑합니다.
    루트(Root)와 개별 프로젝트의 수많은 의존성을 단일 `uv.lock` 파일로 통합 관리하면서도,
    실행 시에는 런타임 환경을 안전하게 격리할 수 있는 강력한 워크스페이스 기능을 제공합니다.

### 5.2 코드 품질 (Ruff & MyPy)
*   **Ruff:** Linter와 Formatter가 통합된 초고속 정적 분석 도구입니다.
    프로젝트 전체의 코드 스타일을 일관되게 강제하여 유지보수성을 극대화합니다.
*   **MyPy (Type Check):**
    *   **타입 안정성:** Python 고유의 동적 타이핑으로 인해 발생할 수 있는 런타임 에러를 빌드 타임에 사전에 방지합니다.
    *   **에이전트 협업 최적화:** 코드 내 명확하게 선언된 타입 힌트(Type Hint)는 AI 에이전트가 기존 코드를 분석하거나
        새로운 기능을 구현할 때, 시스템의 인터페이스를 빠르고 정확하게 파악하도록 돕는 가장 핵심적인 지침서 역할을 합니다.
