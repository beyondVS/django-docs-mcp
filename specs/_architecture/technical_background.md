# 기술적 배경 및 핵심 개념 (Technical Background & Concepts)

본 문서는 Django Docs MCP 프로젝트의 설계 및 구현에 적용된 주요 기술적 원리와 RAG(Retrieval-Augmented Generation) 시스템의 핵심 아키텍처를 설명합니다. 시스템이 직면했던 제약 사항과 문제를 이해하고, 향후 아키텍처 확장 및 고도화 시 의사결정의 기초 지식으로 활용하기 위해 작성되었습니다.

---

## 1. 검색 모델 아키텍처

현대적인 엔터프라이즈 RAG 시스템은 컴퓨팅 자원의 한계와 검색 정확도 사이의 트레이드오프를 극복하기 위해, 모델의 패러다임별 장점을 취합한 **2단계(2-Stage) 검색 파이프라인**을 구성합니다.

### 1.1 모델 상호작용 패러다임 (Interaction Paradigms)

자연어 쿼리와 문서 간의 유사도를 측정할 때, 신경망이 두 텍스트의 문맥을 교환(Interaction)하는 시점에 따라 아키텍처가 분류됩니다.

| 패러다임 | 대표 모델 | 특징 및 원리 | RAG 파이프라인에서의 역할 |
| :--- | :--- | :--- | :--- |
| **Representation-focused**<br>(독립 표현) | Bi-Encoder<br>*(BGE-m3 등)* | 쿼리와 문서를 독립적으로 인코딩하여 단일 임베딩 벡터로 변환. 사전 인덱싱이 가능해 속도가 매우 빠름. (단점: 긴 문서 정보 손실) | **1차 검색기 (Retriever)**<br>전체 DB에서 빠르게 후보군 추출 |
| **Early Interaction**<br>(초기 상호작용) | Cross-Encoder | 쿼리와 문서를 하나로 이어 붙여 전 층에서 교차 연산. 정확도는 최고이나 연산 비용이 극도로 높음. | **제한적 2차 정렬**<br>실시간 검색 불가, 소수(Top 10) 검증용 |
| **Late Interaction**<br>(지연 상호작용) | ColBERT | 단어 단위의 '다중 벡터'로 인코딩 후, 마지막 단계에서 가벼운 행렬 내적 연산 수행. 지연 병목 해결. | **고성능 2차 리랭커 (Reranker)**<br>Cross-Encoder급 정확도 및 빠른 속도 |

### 1.2 2-Stage 검색 파이프라인 작동 원리

1. **1차 검색 (Retrieval - 후보군 추출)**
   * **목표:** 재현율(Recall) 극대화
   * **방식:** 가벼운 단일 임베딩 벡터와 BM25 키워드 매칭을 결합하여, 전체 데이터베이스를 초고속으로 스캔하고 정답이 포함될 후보(Top-K)를 넓게 추출합니다.
2. **2차 정렬 (Reranking - 정밀 검증)**
   * **목표:** 정밀도(Precision) 극대화
   * **방식:** 1차에서 추출된 소수의 문서를 지연 상호작용 모델(ColBERT)에 통과시킵니다. 토큰 단위의 정밀한 문맥 교차 검증을 통해 쿼리 의도에 가장 부합하는 최종 순위를 도출합니다.

## 2. 하이브리드 검색 및 임베딩 최적화

### 2.1 하이브리드 검색 (Hybrid Search)

검색 시 쿼리는 단순 키워드가 아닌 **자연어 형태**를 유지해야 임베딩 모델이 풍부한 문맥을 활용할 수 있습니다. 단일 엔진의 한계를 보완하기 위해 두 방식을 결합합니다.

* **키워드 검색 (BM25 / Lexical Search):** 단어 출현 빈도 기반. 고유 명사, 에러 코드, 메서드명 등 '정확한 매칭'에 매우 강합니다.
* **벡터 검색 (Dense / Semantic Search):** 의미적 유사성 기반. 질문 의도는 같으나 표현 방식이 다른 문서 매칭에 강합니다.

**점수 융합 알고리즘 (Fusion Algorithms)**

| 알고리즘 | 설명 | 적용 권장 사례 |
| :--- | :--- | :--- |
| **RRF**<br>*(Reciprocal Rank Fusion)* | 절대적인 점수를 무시하고 오직 '순위(Rank)'만을 기준으로 공평하게 융합하는 방어적 알고리즘. | 엔진 간 점수 편차가 크고, 균형 잡힌 범용 검색이 필요할 때 |
| **가중치 기반 선형 결합**<br>*(Weighted Linear)* | 두 점수를 0~1로 정규화한 뒤 명시적 가중치(예: Vector 0.3 / BM25 0.7)를 부여하여 통제하는 방식. | 특정 고유명사나 코드 매칭이 중요한 도메인 (기술 문서 검색 등) |

### 2.2 임베딩 차원(Dimension) 전략

* **차원의 개념:** 텍스트의 의미를 압축한 실수 배열의 길이(예: 1024차원)입니다. 고차원일수록 정밀하지만 스토리지와 연산 비용이 비례하여 증가합니다.
* **MRL (Matryoshka Representation Learning):** 텍스트의 핵심 정보가 벡터의 앞쪽 차원에 집중되도록 훈련하는 최신 기술입니다. 시스템의 인프라 제약(스토리지 용량, 메모리 한계)에 맞춰 단일 모델의 출력을 원하는 길이로 잘라서(Truncation) 유연하게 DB 스키마를 설계할 수 있습니다.

## 3. 인프라스트럭처 및 시스템 설계

### 3.1 벡터 데이터베이스 아키텍처

| 데이터베이스 | 특성 및 한계 | RAG 시스템 적용 판단 |
| :--- | :--- | :--- |
| **PostgreSQL**<br>*(pg_vector)* | 실시간 상태 변경(ACID)이 필수적인 쓰기 중심(Write-heavy) 워크로드에 특화. | **(주의)** 1차 검색용으로는 우수하나, Reranker용 다중 벡터 저장 및 연산 시 심각한 캐시 오염 및 디스크 풀 스캔 유발. |
| **LanceDB** | 주기적 업데이트가 발생하는 읽기 중심(Read-heavy) 워크로드에 특화. | **(권장)** RAG 지식 베이스용. 다중 벡터 저장 및 BM25 인덱싱을 단일 스토리지에서 고속 네이티브 지원. |

### 3.2 하드웨어 연산 아키텍처

* **GPU 인프라 환경 (컴퓨팅 중심):** `pg_vector`를 스토리지로 가볍게 유지하고, 원본 텍스트를 물리적으로 분리된 외부 GPU 추론 서버(TEI 등)로 넘겨 실시간(On-the-fly) ColBERT 연산을 수행합니다.
* **순수 CPU 환경 (스토리지 중심):** 무거운 실시간 벡터 연산이 불가하므로, 오프라인 인덱싱 시점에 다중 벡터를 미리 생성하여 LanceDB에 저장합니다. 런타임에는 mmap(Memory-mapped) 기반의 가벼운 디스크 탐색으로 응답 속도를 방어합니다.
* **양자화 최적화 (Quantization - INT8):** CPU 환경 시 모델 가중치를 8비트 정수로 변환합니다. 정밀도 손실을 약간 감수하더라도 메모리 사용량을 대폭 줄이고 정수 연산 가속을 통해 추론 속도를 비약적으로 향상시킵니다.

## 4. 데이터 파이프라인 (수집 및 청킹)

### 4.1 수집 및 전처리 (Ingestion)
*   **전략적 HTML 크롤링:** 소스 코드(RST) 변환 시 발생하는 링크 누락 및 구조 왜곡을 방지하기 위해, 렌더링된 공식 웹사이트를 직접 크롤링합니다. 이를 통해 100% 유효한 절대 링크와 완전한 문서 계층을 확보합니다.
*   **정밀 본문 식별 (Selector-based Extraction):** 일반적인 Readability 알고리즘 대신, 대상 사이트의 구조를 분석하여 기술 문서 본문이 담긴 특정 HTML 요소(예: `<article id="docs-content">`, `[role="main"]`)를 정밀하게 추출합니다. 이는 내비게이션, 사이드바 등 RAG 노이즈를 근본적으로 제거합니다.
*   **공통 수집 엔진 (BaseCrawler):** `asyncio.Queue` 기반의 재귀적 크롤링 로직을 추상화하여, 새로운 문서 소스 추가 시 설정값 정의만으로 고성능 수집 파이프라인을 즉시 가동할 수 있는 구조를 갖추었습니다.
*   **지수 백오프 (Exponential Backoff):** 크롤링 시 대상 서버의 요청 실패(429 에러 등) 발생 시 재시도 간격을 점진적으로 늘려 타겟 서버 부하를 막고 수집 성공률을 높입니다.

### 4.2 의미론적 청킹 (Semantic Chunking)
* **Markdown 구조 활용:** 헤더(H1~H3) 정보를 메타데이터로 유지하여, 파편화된 문서 청크만으로도 해당 청크가 속한 논리적 주제(Section) 문맥을 파악할 수 있도록 돕습니다.
* **코드 블록 보호:** 기술 문서의 핵심 가치인 예제 코드의 무결성을 위해, 문서 분할 시 어떠한 경우라도 코드 블록(` ``` `) 내부는 강제로 분리하지 않습니다.

## 5. 시스템 제약 및 구현 최적화

### 5.1 RAG 컨텍스트 제약 (Context Limits)

> ⚠️ **설계 핵심 지침: 70~80% 제한 룰**
> 대규모 언어 모델은 입력 가능한 최대 토큰(Token) 수가 엄격히 제한됩니다. 특히 리랭커 모델의 최대 컨텍스트(예: 512 토큰)는 **'사용자 질문'과 '문서 청크'를 합친 전체 크기**를 의미합니다.
>
> 리랭킹 과정에서 문서의 꼬리 부분이 잘려나가는 정보 누락(Truncation)을 막기 위해, 문서 청크 크기는 리랭커 최대 한계의 약 70~80% 수준으로 안전하게 제한하여 설계해야 합니다.

### 5.2 비동기 아키텍처 (Full-path Async)
사용자 요청 수신부터 데이터베이스 조회, 임베딩 API 호출, 최종 에이전트 응답에 이르는 전 구간을 논블로킹(Non-blocking) 비동기 파이프라인으로 구축합니다. I/O बा운드 작업 시 스레드를 차단하지 않기 때문에, 적은 서버 리소스로도 다수의 동시 요청을 처리할 수 있는 높은 시스템 동시성(Concurrency)을 확보합니다.
