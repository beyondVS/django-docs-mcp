# 기술적 배경 및 핵심 개념 (Technical Background & Concepts)

본 문서는 Django Docs MCP 프로젝트의 설계 및 구현에 적용된 주요 기술적 원리와 개념들을 설명합니다. 이는 시스템의 제약 사항을 이해하고, 향후 아키텍처 확장 시 의사결정의 기초 지식으로 활용됩니다.

## 1. 하이브리드 검색 (Hybrid Search)

단일 검색 엔진의 한계를 극복하기 위해 두 가지 상이한 검색 방식을 결합합니다.

*   **키워드 검색 (BM25/Lexical Search)**:
    *   **원리**: 단어의 출현 빈도(TF)와 역문서 빈도(IDF)를 기반으로 점수를 산출합니다.
    *   **특징**: 고유 명사나 특정 메서드명 등 '정확한 매칭'이 필요한 경우에 강력한 변별력을 가집니다.
*   **벡터 검색 (Dense Retrieval/Semantic Search)**:
    *   **원리**: 문장을 고차원 벡터로 변환하여 '의미적 유사성(Cosine Similarity)'을 계산합니다.
    *   **특징**: 질문의 의도가 유사하지만 표현이 다른 경우에도 관련 문서를 찾아낼 수 있습니다.
*   **RRF (Reciprocal Rank Fusion)**:
    *   **원리**: BM25 점수(0~10+)와 벡터 점수(0~1)는 스케일이 달라 직접 합산이 불가능하므로, 오직 '순위(Rank)' 정보를 활용하여 `1 / (rank + k)` 공식을 적용해 두 검색 결과를 융합합니다.
    *   **이점**: 개별 검색 엔진의 점수 편향에 상관없이 양쪽에서 공통적으로 상위에 오른 문서에 높은 우선순위를 부여합니다.

## 2. 수집 및 정제 기술 (Ingestion & Cleaning)

### 2.1 본문 추출: Readability 알고리즘
*   **원리**: HTML 요소의 밀도, 텍스트 길이, 링크 밀도 등을 분석하여 페이지의 핵심 본문 영역을 식별하는 휴리스틱 알고리즘입니다. 이를 통해 메뉴, 광고 등 노이즈를 배제한 순수 기술 문서 본문만을 추출합니다.

### 2.2 지수 백오프 (Exponential Backoff)
*   **원리**: 서버 요청 실패 시 재시도 간격을 2^n 단위로 늘려가는 방식입니다. 429(Too Many Requests) 에러 발생 시 서버 부하를 줄이면서 수집 성공률을 높이는 표준적인 전략입니다.

## 3. RAG 파이프라인의 핵심 제약 (RAG Constraints)

### 3.1 토큰(Tokens)과 컨텍스트 한계 (Context Limits)
*   **토큰**: 모델이 이해하는 최소 의미 단위입니다.
*   **제약 사항**: 모델마다 입력 가능한 최대 토큰 수가 엄격히 제한되어 있습니다. 특히 리랭커 모델의 한계(예: 512 토큰)는 질문과 문서 청크를 합친 크기이므로, 문서 청크 크기 결정 시 가장 중요한 제약 조건이 됩니다.
*   **설계 지침**: 리랭킹 과정에서의 정보 누락(Truncation)을 방지하기 위해 청크 크기는 리랭커 컨텍스트의 약 70~80% 수준으로 유지하는 것이 권장됩니다.

### 3.2 리트리벌(Recall) vs 리랭킹(Precision)
*   **Retrieval (재현율)**: 수만 개의 청크 중 관련 후보를 빠르게 뽑아내는 단계입니다. (속도 중시)
*   **Reranking (정밀도)**: 소수의 후보를 대상으로 관련성을 정밀하게 재평가합니다. (정확도 중시)
*   **설계 지침**: 1차 검색(Retrieval)은 최대한 많은 정답 후보를 포함해야 하며, 2차 리랭킹은 그 후보들 사이의 순위를 교정하는 데 집중해야 합니다.

## 4. 모델 및 인프라 최적화 (Optimization)

### 4.1 양자화 (Quantization): INT8
*   **원리**: 모델 가중치를 8비트 정수로 변환하여 메모리 사용량을 1/4로 줄이고 CPU의 정수 연산 가속을 활용하는 기법입니다.
*   **특징**: 미세한 정밀도 손실이 발생할 수 있으나, CPU 환경에서의 추론 속도를 비약적으로 향상시킬 수 있어 실용적인 트레이드오프가 성립합니다.

### 4.2 비동기 아키텍처 (Full-path Async)
*   **원리**: I/O 바운드 작업 시 스레드를 차단하지 않고 다른 작업을 처리하는 방식입니다. DB 조회부터 에이전트 응답까지 전구간 비동기 파이프라인을 구축하여 높은 동시성을 확보하는 것이 목표입니다.

## 5. 의미론적 청킹 (Semantic Chunking)

*   **Markdown 구조 활용**: 헤더(H1~H3) 정보를 활용해 각 청크가 어떤 논리적 주제에 속하는지 메타데이터를 유지하여 검색 시 문맥 이해도를 높입니다.
*   **코드 블록 보호**: 기술 문서의 핵심인 예제 코드의 무결성을 위해, 코드 블록(` ``` `) 내부는 절대 분리하지 않는 것을 원칙으로 합니다.
