**기능 브랜치**: `003-improve-markdown-chunking`
**생성일**: 2026년 3월 18일
**상태**: 초안(Draft)
**입력**: 사용자 설명: "청킹 전략을 코드 블럭이 잘리지 않도록 변경. # 1. 핵심 요구사항 - 마크다운 문서 내의 코드 블록(```)이 중간에 잘리는 것을 최대한 방지해야 합니다. - 최종 청크의 크기는 임베딩 모델에 적합하게 제한되어야 합니다. # 2. 사용 기술 및 클래스 (langchain-text-splitters) 반드시 다음 두 가지 클래스를 직렬(Chaining)로 연결하여 2단계 파이프라인으로 구현하십시오. - Phase 1: MarkdownHeaderTextSplitter - Phase 2: MarkdownTextSplitter # 3. 구현 제약사항 - Document 객체 리스트 입출력, 오버랩 시 코드 블록 보호, 순수 텍스트 유지."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 문서 논리 구조 및 문맥 보존 (우선순위: P1)

사용자는 긴 마크다운 문서를 검색할 때, 특정 섹션의 내용이 어떤 헤더(Header 1, 2, 3) 아래에 속해 있었는지에 대한 문맥 정보를 함께 얻기를 원합니다.

**우선순위 이유**: 검색 결과의 정확도와 답변 생성 시의 문맥 이해를 위해 문서의 구조 정보는 필수적입니다.

**독립적 테스트**: 헤더가 포함된 마크다운 문서를 입력으로 넣었을 때, 분할된 각 청크의 메타데이터에 해당 섹션의 헤더 정보가 올바르게 포함되어 있는지 확인하여 테스트할 수 있습니다.

**수락 시나리오(Acceptance Scenarios)**:

1. **Given** 여러 계층의 헤더(#, ##, ###)가 포함된 마크다운 문서가 주어졌을 때, **When** 청킹 프로세스를 실행하면, **Then** 각 청크는 자신이 속한 헤더의 텍스트를 메타데이터(예: Header 1, Header 2 등)로 가지고 있어야 합니다.
2. **Given** 문서가 헤더를 기준으로 분할되었을 때, **When** 특정 헤더 아래의 내용이 너무 작으면, **Then** 해당 내용은 독립적인 청크로 유지되거나 논리적으로 결합되어야 합니다.

---

### 사용자 스토리 2 - 코드 블록 무결성 보장 (우선순위: P1)

개발 사용자는 기술 문서 검색 시 코드 예제가 중간에 잘리지 않고 온전한 형태로 제공되기를 원합니다. 특히 코드 블록(```) 내부의 내용이 청킹 과정에서 분리되어 의미가 훼손되는 것을 방지해야 합니다.

**우선순위 이유**: 코드 블록이 잘리면 코드의 가독성이 떨어질 뿐만 아니라, AI가 잘못된 코드를 생성하거나 해석할 위험이 큽니다.

**독립적 테스트**: 긴 코드 블록이 포함된 문서를 입력으로 넣고, 결과 청크들 중에서 코드 블록의 시작(```)과 끝(```)이 한 청크 내에 존재하거나, 코드 블록 내부에서 강제로 잘린 지점이 없는지 검증합니다.

**수락 시나리오**:

1. **Given** 텍스트 중간에 코드 블록(```...```)이 포함된 문서가 있을 때, **When** 2단계 청킹 파이프라인을 통과하면, **Then** 코드 블록의 시작부와 종료부는 반드시 동일한 청크 내에 존재해야 합니다.
2. **Given** 매우 긴 코드 블록이 있어 부득이하게 분할이 필요한 상황이더라도(예외적 상황), **When** 분할이 발생한다면, **Then** 마크다운 문법을 존중하여 코드의 의미가 보존되는 지점에서 분할되어야 하며 오버랩으로 인해 코드 블록 기호가 깨지면 안 됩니다.

---

### 사용자 스토리 3 - 임베딩 최적화 크기 조절 (우선순위: P2)

시스템은 검색 효율을 위해 너무 큰 청크를 임베딩 모델이 처리하기 적절한 크기로 재분할해야 합니다. 이때 1단계에서 추출된 헤더 문맥은 유지되어야 합니다.

**우선순위 이유**: 임베딩 모델의 토큰 제한을 초과하는 청크는 정보 손실이 발생하므로, 안전한 재분할이 필요합니다.

**독립적 테스트**: 1단계(헤더 분할) 결과물 중 임계값(예: 800 토큰)을 초과하는 대형 청크가 2단계(텍스트 분할)를 거친 후, 설정된 크기 이하의 여러 청크로 나뉘면서도 동일한 헤더 메타데이터를 유지하는지 확인합니다.

**수락 시나리오**:

1. **Given** 1단계 분할 결과 청크가 임계 크기를 초과할 때, **When** 2단계 파이프라인이 실행되면, **Then** 최종 결과물은 임계 크기 이하의 여러 Document 객체로 반환되어야 합니다.
2. **Given** 재분할된 여러 개의 청크가 생성되었을 때, **When** 각 청크를 확인하면, **Then** 모든 청크는 1단계에서 부여된 동일한 헤더 정보를 메타데이터로 공유해야 합니다.

### 예외 케이스 (Edge Cases)

- **헤더가 없는 문서**: 문서 전체가 하나의 청크로 처리된 후 2단계에서 크기 기준으로만 분할되는지 확인합니다.
- **매우 빈번한 헤더**: 헤더가 너무 많아 청크 크기가 극단적으로 작은 경우(예: 10자 미만)의 처리 방식을 정의해야 합니다.
- **중첩된 코드 블록**: 코드 블록 내부에 다른 마크다운 문법이 섞여 있을 때의 안전한 분할 여부를 확인합니다.

## 요구 사항 *(필수)*

### 기능적 요구 사항 (Functional Requirements)

- **FR-001 (2단계 파이프라인)**: 시스템은 반드시 `MarkdownHeaderTextSplitter`(Phase 1)와 `MarkdownTextSplitter`(Phase 2)를 직렬로 연결한 파이프라인을 구현해야 함.
- **FR-002 (헤더 기반 분할)**: Phase 1은 반드시 `#`, `##`, `###` 헤더를 기준으로 문서를 분할하고, 이를 `Header 1`, `Header 2`, `Header 3` 메타데이터 필드에 저장해야 함.
- **FR-003 (코드 블록 보호)**: Phase 2는 재분할 시 마크다운 문법을 인식하여 코드 블록(```) 내부가 잘리는 것을 최대한 방지해야 함.
- **FR-004 (기존 인터페이스 유지)**: 시스템은 기존 `ChunkingService.split_markdown()` 메서드의 인터페이스(`str` 입력, `list[dict[str, Any]]` 반환)를 유지해야 함. 내부 파이프라인 구현만 변경하며, 외부 호출부에 영향을 주지 않아야 함.
- **FR-005 (데이터 무결성)**: 청킹된 텍스트 본문에는 헤더 정보나 파일명 등 외부 정보를 직접 삽입하지 않으며, 모든 부가 정보는 `Document.metadata` 필드를 통해서만 관리해야 함.
- **FR-006 (오버랩 제약)**: 재분할 시 오버랩(Overlap) 설정으로 인해 코드 블록의 시작/종료 기호가 중복되거나 잘려나가는 부작용이 없어야 함.

### 시스템 제약 사항 (System Constraints - Django Docs MCP)

- **SYS-001**: 최종 청크의 크기는 문자 수 기반으로 측정하며, 임베딩 모델의 토큰 제한을 고려하여 약 1500~2500자 범위를 목표로 함 (토큰 ≈ 문자수/3~4 비율 환산 기준). 구체적인 설정값은 Phase 2의 `chunk_size` 파라미터로 조절.
- **SYS-002**: 기존 `django_server` 내의 `ChunkingService` 인터페이스와 호환되어야 함.
- **SYS-003**: 파이썬 코드 블록은 마크다운 내에서 ` ```python `으로 명시된 경우 특히 보호되어야 함.

### 주요 엔티티

- **ChunkPipeline**: 1단계와 2단계 스플리터를 관리하고 Document 리스트를 처리하는 실행 단위.
- **EnhancedDocument**: 본문 텍스트와 함께 계층적 헤더 정보(H1, H2, H3)를 포함하는 메타데이터를 가진 객체.

## 성공 기준 *(필수)*

### 측정 가능한 결과 (Measurable Outcomes)

- **SC-001**: 테스트 세트의 마크다운 문서 중 코드 블록이 포함된 청크의 100%가 문법적으로 유효한(시작과 끝이 매칭되는) 코드 블록을 유지해야 함.
- **SC-002**: 모든 최종 청크는 최소 하나 이상의 상위 계층 헤더 정보를 메타데이터로 포함해야 함 (문서 상단에 헤더가 존재하는 경우).
- **SC-003**: 생성된 모든 청크의 크기는 설정된 `chunk_size` 상한값을 초과하지 않아야 함.
- **SC-004**: 기존 단일 단계 청킹 방식 대비, 코드 블록 내 절단 발생 빈도가 0건으로 감소해야 함.

## Clarifications

### Session 2026-03-18

- Q: 기존 `split_markdown()` 인터페이스(`dict` 반환)를 유지할 것인가, `Document` 객체 기반으로 교체할 것인가? → A: 기존 `dict` 기반 인터페이스를 유지한다. `Document` 객체 리스트 반환 요구사항은 취소.
- Q: 청크 본문(content)에 `Context:`, `Document:` 등의 컨텍스트를 직접 주입하는 기존 동작을 유지할 것인가? → A: FR-005대로 본문 내 컨텍스트 주입을 제거하고, 모든 부가 정보는 메타데이터로만 관리한다.
- Q: `chunk_size` 측정 단위를 토큰 기반으로 할 것인가, 문자 수 기반으로 할 것인가? → A: 문자 수 기반을 유지한다. 토큰 대비 문자수 비율(약 1:3~4)로 환산한 적정 상한값을 설정하며, 별도 토크나이저 의존성은 추가하지 않는다.
