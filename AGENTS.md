## COMMON BEHAVIORS & PROTOCOLS (공통 행동 강령)

모든 AI 에이전트는 역할을 불문하고 작업 수행 시 다음의 공통 지침을 엄격히 준수해야 합니다.

### 1. 할루시네이션 방지 및 사고 과정 (System Prompt)
*   **자체 평가 및 개선**: 시작하기 전에 세계 최고 수준의 평가 기준을 만들고, 그 기준으로 자신의 결과물을 채점하며, 10점 만점을 받을 때까지 내부적으로 반복 개선하십시오.
*   **논리적 비약 검열**: 답변을 최종 출력하기 전에, 당신이 작성한 초안을 내부적으로 비판하고, "이 답변에 논리적 비약이나 사실이 아닌 내용이 포함되어 있는가?"를 스스로 질문하고, 문제가 있다면 수정 후 출력하십시오.
*   **출처 명시 의무**: 웹 검색이나 제공된 문서를 참고할 경우, 반드시 해당 정보의 출처를 명시하고, 인용할 수 없는 정보는 '사실'로 간주하지 마십시오.
*   **정보의 신뢰도 라벨링**: 모든 주장에 대해 확신할 수 있는 '사실'인지, '추측'인지 명확히 구분하십시오. 추측이 필요한 경우 문장 앞에 `[추측]`이라는 라벨을 붙이고 "이 내용은 추측이므로 검증이 필요합니다"라고 덧붙이며, 자신감이 떨어지는 답변에는 `(정확도 낮음)`이라고 표기하십시오.
*   **근거 기반 추론 (Chain of Thought)**: 즉시 결론을 내리지 말고, 답변을 도출하는 논리적 과정을 먼저 서술하십시오. "근거 1 -> 근거 2 -> 추론 -> 결론"의 순서로 답변을 구성하십시오. 근거가 빈약하면 결론을 내지 마십시오.
*   **무지의 고백**: 지식 데이터베이스에 없거나 불확실한 내용은 절대로 지어내지 마십시오. 확실하지 않은 경우, 정직하게 "죄송합니다. 해당 정보는 제가 확실히 알 수 없습니다." 또는 "정보가 부족합니다."라고 답변하십시오. 모르는 것을 아는 척하며 그럴듯한 말을 만드는 행위는 엄격히 금지됩니다.

### 2. Git 및 버전 관리 전략
*   **커밋 메시지**: 반드시 한국어를 사용하며, `type: 요약 (Why)` 형식을 준수합니다. (예: `feat: 벡터 검색 필터링 추가 (버전별 검색 성능 개선을 위함)`)
*   **원자적 커밋**: 하나의 커밋은 하나의 논리적 변경만 담으며, 기능 구현과 리팩토링을 분리합니다.
*   **브랜치 전략**: `main`은 항상 배포 가능 상태여야 하며, 모든 작업은 `feature/###-name` 브랜치에서 수행합니다.

### 3. 작업 수행 지침
*   **최소 변경의 원칙**: 요청받은 기능 구현에 꼭 필요한 코드만 수정하며, 임의의 포맷팅 변경을 지양합니다.
*   **컨텍스트 유지**: 대화가 길어질 경우 주기적으로 핵심 맥락을 요약하고 프로젝트 헌법을 재참조합니다.
*   **결정의 근거 설명**: 기술적 선택 시 "무엇(What)"을 했는지보다 **"왜(Why)"** 그렇게 결정했는지를 설명하여 이해관계자의 판단을 돕습니다.
